## 1. 공간 쿼리 — 어디서 필터링할 것인가

**상황**
트래픽의 대부분이 두 쿼리다 — "내 위치 반경 N km 안의 제보" 와 "지도 viewport 안의 제보". 정확도와 응답 시간이 서비스 체감을 직접 좌우한다.

**처음 접근의 한계**
초기에는 Supabase REST로 `reports`를 다량 받아 와서 **Python에서 Haversine 거리를 계산**했다 (`benchmark_nearby_rest_python`로 비교용 함수가 코드에 남아있다). 데이터가 늘수록 네트워크 페이로드와 애플리케이션 CPU가 같이 증가하는 구조였다.

**검토한 대안**
- **Geohash 버킷팅** — 구현은 쉽지만 반경 경계 박스 처리가 복잡하고, "거리순"이나 "가중치 기반 정렬" 같은 확장 요구에 약하다.
- **클라이언트 거리 필터 유지** — 응답 페이로드 자체가 줄지 않는다.
- **PostGIS + GiST 인덱스** — 학습·마이그레이션 비용은 크지만, BBox 1차 필터 + 정확 거리 2차 필터의 2단계 구조로 인덱스를 안정적으로 활용할 수 있다.

PostGIS를 택했다. **정확도와 정렬 확장성을 우선**시한 결정. 향후 "거리·인기·신선도 가중치" 같은 정렬 요구를 DB가 처리하게 두는 게 유리하다고 봤다.

**운영에서 만난 문제**
- 단순히 함수만 만들었더니 RLS 오버헤드와 함수 호출 비용 때문에 인덱스가 일관되게 활용되지 않았다. **함수 속성을 STABLE로 선언 + 적절한 권한 설정**으로 옵티마이저가 인라이닝하도록 조정한 뒤에야 의도한 실행 계획이 나왔다.
- 이후 vote/comment count를 응답에 합치면서 **함수 반환 타입이 바뀌었고, PostgreSQL 함수 오버로딩 충돌**을 겪었다. 이를 계기로 마이그레이션마다 **기존 함수 명시적 제거 → 재생성** 패턴을 표준화했다.

**최종 trade-off와 측정 결과**
Locust 부하 테스트는 2차에 걸쳐 진행됐다. **1차(단일 워커, 50 users)**: 모든 엔드포인트가 p50 20~25s로 평준화돼 V1/V3 DB 단 차이가 큐잉에 가려졌다 — 셋업이 DB 비교가 아니라 백엔드 동시성을 측정하고 있었다는 진단. **2차(워커 4, 20 users)**: 큐잉을 해소하자 가설이 검증됐다 — **V3가 V1 대비 p50 −27%(1,500→1,100ms), p99 −9%, RPS +15%**. 양쪽 모두 실패율 0%로 안정성도 일관 유지. 절대 응답 시간(p50 ≈ 1s)에는 클라우드 RTT가 베이스라인으로 남아있어, DB 호출 구간 분리 계측이 다음 단계. 자세한 수치·셋업·v1↔v2 비교는 [LOAD_TEST_RESULTS.md](./LOAD_TEST_RESULTS.md). 측정 비용 외에도 **로컬 환경 복잡도와 함수 시그니처 관리 비용**이 함께 늘었다.

> 데이터 필터링은 애플리케이션이 아니라 DB에서, 인덱스를 활용하는 형태로. 그리고 부하 테스트는 "측정하려던 것"이 아니라 "환경에서 가장 좁은 병목"을 드러낸다 — 셋업을 한 번 교정하기 전엔 결과를 믿지 말 것.

## 2. N+1 — 리스트 1번 + 카운트 2N번

**상황**
페이지당 9개 제보 리스트인데, 카드마다 vote 수·comment 수·"내가 투표했는지"가 필요했다.

**처음 접근의 한계**
리스트 1번 + 카드마다 카운트 2번 = 1+2N 쿼리. 어드민 대시보드는 더 심해서 통계 12개를 각각 호출했다.

**검토한 대안**
- **nested select**: 단건 조회엔 좋지만 리스트엔 row마다 nested, RLS와 섞이면 디버깅이 어려워진다.
- **Materialized View**: write가 잦은 데이터라 갱신 빈도 vs 정확성 trade-off가 컸다.
- **DB 함수 내부에서 카운트 집계**: 쿼리 1번. 카운트 대상 row 수가 작아 비용이 크지 않다.

세 번째를 택했다. `get_reports_paginated`, `get_admin_dashboard_stats`, `get_profile_with_stats` 모두 같은 패턴.

**운영에서 만난 문제**
"내가 투표했나" 정보는 사용자 컨텍스트를 함수 안에서 깔끔하게 다루기 어려웠다. 결국 **서비스 레이어에서 batch lookup(쿼리 1번)으로 합치는 하이브리드**가 됐다. 완벽한 분리가 아니라 **현실적 절충**.

**최종 trade-off**
쿼리 횟수가 1+2N → 2~3으로 줄었지만, DB 함수가 비즈니스 로직을 일부 들고 있게 됐다. 앱 코드와 DB 함수 양쪽을 같이 봐야 흐름이 잡힌다.

> N+1은 ORM이 아니라 쿼리 설계의 문제.

## 3. 대량 마커 렌더링

**상황**
지도 zoom out 시 마커가 수백 개로 늘었고, 단순 DOM 마커로는 zoom·pan 한 번에 프레임 드롭이 명확히 느껴졌다.

**처음 접근의 한계**
SDK 기본 마커를 `reports.map()` 으로 그렸다. zoom이 바뀌면 마커가 통째로 다시 그려졌고, 상위 hook이 fetch마다 새 배열 참조를 만들어서 메모이제이션이 무효화됐다.

**검토한 대안**
- **Canvas overlay로 직접 그리기**: 가장 빠르지만 클러스터 UI·툴팁·클릭 핸들링을 다 직접 만들어야 한다. 비용 과다.
- **viewport 가상화만**: 도심 밀집 지역에선 여전히 수십 개 동시 노출.
- **SDK 클러스터러 + viewport 밖 마커 미렌더 + React 18 deferred 렌더링**.

세 번째. **직접 Canvas를 피한 이유는 클러스터 클릭·줌·인포윈도우 같은 UX를 SDK가 이미 제공**하기 때문. 직접 만들면 새 버그 표면을 늘리는 것.

**운영에서 만난 문제**
가장 큰 함정은 **참조 동일성**(reference identity)이었다. useMemo로 안정화해도 상위 hook이 fetch마다 새 배열을 주면 무의미했다. React Query staleTime + refetch 정책으로 데이터 소스에서부터 참조를 안정화해야 했다. 추가로 zoom 레벨별 동적 limit(zoom ≥8: 500, ≥6: 300, ≥4: 200, else 100)으로 서버 응답 크기도 줄였다.

**최종 trade-off**
SDK 종속성이 늘었고 클러스터 스타일 커스터마이징은 SDK 옵션 안에서만 가능. 대신 **버그 표면적과 유지보수 비용**이 줄었다.

> 렌더링 성능은 UI 라이브러리가 아니라 데이터 흐름 + 뷰포트 인지 + 스케줄링의 합으로 결정된다.

## 4. 권한 — 코드 인가 로직을 데이터 정책으로

**상황**
"내 제보만 수정", "투표는 로그인만", "어드민 로그는 서비스 권한만" 같은 규칙이 처음엔 라우터·서비스 코드 if 문에 흩어져 있었다.

**처음 접근의 한계**
같은 정책이 여러 경로에 중복돼서, 정책 변경 시 누락 위험이 컸다. 실제로 vote 삭제 ownership 체크를 한 경로에선 했는데 다른 경로에선 빠뜨린 적이 있다.

**검토한 대안**
- **서버 미들웨어로 통일**: 여전히 로직이라 분기가 늘면 또 흩어진다.
- **Supabase RLS로 이전**: 정책이 데이터 옆에 붙어 일관성에 유리.

RLS를 택했다.

**운영에서 만난 문제**
- **Supabase Advisor 경고**: RLS 정책 안에서 사용자 함수를 직접 호출하면 row마다 재평가된다. **한 번만 평가되도록 정책 표현 방식을 일괄 수정**했다. 작은 변경이지만 정책이 많은 테이블일수록 효과가 크다.
- 어드민 로그 테이블은 초기에 RLS만 켜고 정책이 없는 상태였다. 서비스 권한 전용 정책으로 명시적으로 막았다.

**최종 trade-off**
인가가 코드 외부로 빠지면서 **단위 테스트로 검증이 어려워졌다**. 통합 테스트나 실제 RLS 환경에서 봐야 한다. 대신 **누락 위험·중복**이 사라졌다.

> 권한은 "로직"이 아니라 "데이터에 붙은 정책"으로 다뤄야 변경에 강하다.

## 5. 운영 중 갚은 기술 부채 (2026.02–05)

MVP 배포 후 8개월간 누적된 부채를 4가지 축으로 정리했다.

- **서비스 레이어 분해**: 모놀리식 비즈니스 로직을 도메인 단위(report / comment / vote / profile / admin{dashboard, user, report, log})로 분리. 라우터는 thin handler로.
- **Pydantic v1 → v2**: deprecation 경고를 운영 로그에서 발견하고 시작. v2 표준 표기로 일괄 마이그레이션.
- **프론트 클린 아키텍처 이관**: 페이지가 직접 백엔드를 호출하던 구조를 `features/<slice>/{domain, data, presentation}`로 정리. **페이지는 ViewModel hook만 호출**, 데이터 형변환은 데이터 계층 진입점에서 1회.
- **테스트 인프라**: Vitest + RTL + jsdom. ViewModel·Repository 라인 커버리지 80%+ 목표. 번들 분석기로 first-load JS 크기 모니터링.

핵심 판단: **한 번에 다 바꾸지 않는다**. Phase 0.5(boundary 설정) → 1(UI 분리) → 2(Auth) → 3~7(Reports/Map/Admin)로 단계마다 빌드·테스트 그린을 유지하며 머지. **큰 PR 1개보다 작고 검증된 PR 7개가 운영 중 리팩토링에 더 안전했다.**

## 6. AI 협업 — 거부 사례 2개

**케이스 1 · Generic Repository 추상화 거부**
초기 이관 때 AI가 "모든 도메인을 감싸는 generic Repository"를 제안했다. 빠르게 만들 수 있었지만 도메인 차이(vote는 toggle/ownership, report는 spatial 쿼리)가 추상화에 묻혔다. **도메인별 명시적 인터페이스**(ReportRepository, VoteRepository, CommentRepository, ImageRepository)로 다시 설계. **SRP를 지키는 게 추상화의 우아함보다 우선**.

**케이스 2 · 모놀리식 어드민 분해**
한 파일에 어드민 로직을 다 몰아넣은 초기 AI 코드가 운영 중 변경에 약했다. 사용자 관리·제보 관리·대시보드·로그를 동시에 건드리는 PR이 자주 생기고 충돌이 났다. **책임 경계 재정립** 후엔 PR 하나가 한 가지 책임만 건드리게 됐다.

**개발 운영 방식**
- Phase 단위로 작게 쪼개고, 매 단계 빌드·lint·타입 체크·테스트 통과를 머지 조건으로 강제 (`tsc:check`, `lint`, `pytest -q`)
- 리팩토링 계획 문서에 **검토 라운드(1차~6차)** 를 남겨 자기 코드 리뷰 — AI 제안의 일관성과 trade-off를 단계별로 검증

> AI 시대에 인간이 더 잘해야 하는 건 생성이 아니라 받아들일지 거부할지의 판단.

## 7. 한계와 다음 작업

- **DB 호출 구간 분리 계측 미적용**: v2 재측정으로 V3의 우위는 확인했지만 ([결과](./LOAD_TEST_RESULTS.md)), 절대 응답 시간(p50 ≈ 1s) 중 Supabase 클라우드 RTT가 얼마나 차지하는지는 따로 떼지 못했다. 미들웨어 타이밍 또는 `pg_stat_statements`로 분리 계측이 다음 단계.
- **실사용자 트래픽 측정 부재**: 시나리오 기반 부하는 분포 가정의 한계가 있다. 배포 후 실사용자 패턴 분석이 그 다음 단계.
- **Branch coverage 미적용**: 현재 라인 80% 기준이라 분기 누락 위험이 남는다.
- **Realtime 미적용**: 새 제보·댓글 즉시 반영이 가능하지만 RLS·캐시 무효화 동작이 복잡해져 보류.
