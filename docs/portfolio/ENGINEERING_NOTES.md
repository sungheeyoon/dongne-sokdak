## 1. 현재 공간 조회 경계

현재 프론트엔드의 지도와 목록은 `/api/v1/reports/bounds`만 호출합니다.

- 최초 지도 로드와 검색 결과 선택처럼 지도 초점이 명시적으로 바뀔 때 bounds를 커밋합니다.
- 사용자의 드래그·확대/축소는 현재 화면이 마지막 조회 영역과 달라졌다는 dirty 상태만 만듭니다.
- 사용자가 “이 지역 재검색”을 눌렀을 때 새 bounds를 커밋하고 지도와 목록을 함께 갱신합니다.

백엔드의 `/nearby`와 `/benchmark/nearby-rest`는 과거 반경 검색과 구현 비교를 위한 경로로 남아 있지만 현재 제품 UI에서는 사용하지 않습니다. 따라서 현재 성능 성과는 반경 2km/3km 조회가 아니라 bounds 경로의 측정으로 설명합니다.

## 2. PostGIS를 선택한 이유

초기 구현은 많은 위치 데이터를 애플리케이션으로 가져와 Python Haversine 계산으로 필터링했습니다. 이 구조에서는 데이터가 늘수록 네트워크 전송량과 애플리케이션 CPU가 함께 증가합니다.

검토한 선택지는 세 가지였습니다.

- **클라이언트 필터링**: 구현은 단순하지만 전송량을 줄이지 못하고 조회 책임이 UI에 남습니다.
- **Geohash 버킷**: 빠른 후보 축소에는 유리하지만 경계 버킷 처리와 정확한 거리·복합 정렬 확장이 복잡해집니다.
- **PostGIS**: 스키마와 마이그레이션 비용은 들지만 공간 인덱스, 정확한 공간 술어, 정렬·집계 확장을 DB 경계에 모을 수 있습니다.

PostGIS를 선택했고, 현재 제품 흐름은 지도 사각 영역을 GiST 인덱스와 `&&` envelope 조건으로 조회합니다. 과거 반경 조회의 거리순 결과가 아니라 **현재 UI가 실제로 소비하는 bounds**를 최적화·측정 대상으로 삼았습니다.

## 3. “RPC를 합치면 빨라진다”는 가설이 틀렸던 과정

### 첫 변경

기존 bounds 요청은 전체 개수와 페이지를 순차 RPC 두 번으로 조회했습니다. 이를 `get_reports_in_bounds_page` 한 번으로 합쳐 `items`와 `total_count`를 함께 반환하도록 했습니다.

저부하 직접 RPC 측정에서는 왕복 감소 효과가 보였지만, 4 workers·동시 사용자 20명의 API 부하에서는 다음 결과가 나왔습니다.

| 지표 | 2 RPC | 1 RPC |
| --- | ---: | ---: |
| p50 | 8.7초 | 8.6초 |
| p99 | 19초 | 21초 |
| RPS | 2.14 | 2.09 |

왕복 횟수는 줄었지만 동시 부하 개선을 입증하지 못했습니다. 여기서 수치를 성과로 포장하지 않고 실행계획을 다시 확인했습니다.

### 실행계획 진단

대표 강남 bounds에는 전체 10,006건 중 8,039건이 포함됐습니다.

- GiST 인덱스가 공간 후보를 찾는 시간은 약 1ms였습니다.
- vote는 2건, comment는 0건뿐이라 집계 서브쿼리는 주 병목이 아니었습니다.
- 공통화한 `report_matches_filters`가 category/search가 모두 `NULL`인데도 공간 후보마다 호출됐습니다.

| DB 측정 | 공통 함수 | 인라인 술어 | 변화 |
| --- | ---: | ---: | ---: |
| count | 55.8ms | 6.6ms | 88.2% 감소 |
| 전체 SQL 3회 중앙값 | 125.8ms | 16.0ms | 87.3% 감소 |

### 선택

활성 RPC에서만 category/search 조건을 두 번 인라인했습니다. 중복을 줄이는 것보다 PostgreSQL 플래너가 비활성 필터를 제거하고 실제 선택도에 맞는 계획을 세울 수 있는 경계를 우선했습니다.

중복된 두 술어가 서로 달라지는 위험은 SQL 계약 테스트로 막았습니다. 기존 반경 조회와 롤백용 bounds 함수에는 공통 헬퍼가 남아 있으며, 다시 활성화할 경우 별도 실행계획 검증이 필요합니다.

### 동일 조건 재측정

변경 전 RPC를 임시 이름으로 복제하고 `ReportService`의 RPC 이름만 생성자로 주입해 활성 RPC를 되돌리지 않고 기준선을 재현했습니다. 두 RPC의 total count, page size, 첫 항목이 같은지 확인한 뒤 Locust 2.32.10으로 전후 각 3회 측정했습니다. 측정 후 임시 RPC를 삭제하고 함수 부재를 확인했습니다.

| 지표 | 변경 전 3회 중앙값 | 변경 후 3회 중앙값 | 변화 |
| --- | ---: | ---: | ---: |
| p50 | 8.8초 | 7.6초 | 13.6% 감소 |
| p99 | 21초 | 16초 | 23.8% 감소 |
| 평균 | 8.835초 | 7.614초 | 13.8% 감소 |
| RPS | 2.09 | 2.40 | 15.0% 증가 |
| 실패율 | 0% | 0% | 동일 |

자세한 실행 조건과 원본 파일은 [측정 보고서](../../backend/results/locust/BOUNDS_RPC_BENCHMARK_20260724.md)에 있습니다.

## 4. 목록 N+1

### 문제

목록의 각 카드에는 투표 수, 댓글 수, 로그인 사용자의 투표 여부가 필요합니다. 이를 항목별로 조회하면 목록 1회 뒤에 두 카운트 쿼리가 N번 반복되는 `1 + 2N` 구조가 됩니다.

### 선택

- 투표·댓글 수는 `get_reports_paginated` RPC 내부에서 집계합니다.
- 전체 개수는 `count_reports_paginated`로 조회합니다.
- 사용자별 `user_voted`는 페이지의 report id를 모아 한 번의 batch 조회로 합성합니다.

익명 사용자는 2회, 로그인 사용자는 3회의 DB 경계 호출로 끝납니다. Materialized View의 갱신 비용이나 사용자 컨텍스트를 SQL 함수 안에 밀어 넣는 복잡도를 피한 절충입니다.

## 5. 지도 요청·렌더링 계약

지도 성능은 API만 빨라져서는 해결되지 않습니다. 요청 트리거와 렌더링 데이터 양을 함께 제한했습니다.

- `useKakaoMapBounds`: 드래그·줌은 dirty만 변경하고 명시적 커밋만 상위 bounds를 갱신합니다.
- 회귀 테스트: 드래그 10회와 줌 10회 동안 커밋 0회, 재검색 후 최신 bounds 1회 커밋.
- `MapMarkerLayer`: 결정적 500개 입력 중 viewport 안의 80개만 렌더링하는 테스트.
- 표시 단계: 먼 줌은 Kakao 클러스터, 중간 줌은 30m 근접 그룹, 가까운 줌은 개별 마커.
- 큰 마커 배열 갱신은 `useTransition`으로 낮은 우선순위에 둡니다.

“500개가 언제나 80개가 된다”는 일반 성능 수치가 아니라, 화면 밖 420개가 주어진 테스트 픽스처에서 **culling 계약이 유지되는지**를 검증한 결과입니다.

## 6. 구조와 변경 기록

- 프론트엔드는 feature별 `domain / data / presentation`으로 나누고 domain이 repository port를 소유합니다.
- Kakao의 명령형 API 호출은 `KakaoMapAdapter`에 모으고, 선언형 `react-kakao-maps-sdk` 컴포넌트는 presentation에서 사용합니다.
- 백엔드는 Supabase client만 사용하며 service를 클래스와 생성자 주입으로 구성합니다.
- 캐시 무효화, 지도 초점, bounds 재검색, 근접 그룹, 필터 인라인 같은 판단은 ADR로 남깁니다.
- CI는 frontend lint/typecheck/test와 backend pytest를 실행합니다.

## 7. 측정 한계와 다음 조사

- 부하 테스트는 로컬 FastAPI와 공유 Supabase, 합성 데이터, 결정적 bounds 분포를 사용했습니다. 운영 트래픽이나 SLA를 대표하지 않습니다.
- 필터 함수 병목은 제거했지만 포화 구간의 p50이 여전히 7.6초입니다. 애플리케이션 동시성, 원격 DB RTT, 연결 제한을 분리 계측해야 합니다.
- `/nearby`와 과거 benchmark 코드는 현재 제품 경로가 아니므로, 제거 여부를 코드 정리 단계에서 별도로 결정해야 합니다.
