# 영역 조회는 드래그·확대/축소가 아니라 "이 지역 재검색" 클릭으로만 실행한다

지금까지 지도를 드래그하거나 확대/축소할 때마다 영역 조회(Bounds Query)가 자동 실행되어 마커와 하단 리포트 목록이 즉시 다시 그려졌다. 사용자 입장에서는 지도를 조금만 움직여도 화면 전체가 매번 리렌더링되는 것으로 체감되고, 실제로 보고 싶은 건 "지금 보이는 영역"이 아니라 "재검색을 눌렀을 때의 영역"인 경우가 많다. 이 결정은 드래그·확대/축소를 "영역 조회 트리거"에서 제외하고, 지도 위에 뜨는 "이 지역 재검색" 버튼 클릭이라는 명시적 사용자 의도가 있을 때만 영역 조회를 실행하도록 바꾼다.

드래그·확대/축소는 화면상 위치만 추적(dirty 여부)할 뿐 아무것도 커밋하지 않는다 — `useKakaoMapBounds`의 `currentBounds`(뷰포트 컬링 기준)와 store의 `currentMapBounds`(조회 기준)는 마지막으로 커밋된 영역에 그대로 얼어 있고, 화면이 그 영역과 달라지면 `isDirty`만 true가 된다. `isDirty`가 true인 동안 지도 위에 재검색 버튼이 뜨고, 클릭하는 순간 그 시점의 실제 지도 bounds를 커밋해 마커·리스트를 함께 갱신한다.

다음은 여전히 영역 조회를 자동 실행하는 명시적 사용자 의도로 남는다 — [ADR-0003](./0003-map-focus-vs-picked-point-kakao-adapter.md)의 "지도 초점을 바꾸는 명시적 트리거" 목록과 같은 기준이다:
- 최초 지도 로드
- "내 동네로 돌아가기" / 검색 초기화
- 검색 결과(장소) 선택
- 텍스트 검색어 입력, 카테고리 필터 변경 (지도 이동이 아니라 조회 조건 자체의 변경)

## Considered Options

- **드래그만 자동 재조회를 끄고 확대/축소는 그대로 둔다**: 드래그와 확대/축소는 `useKakaoMapBounds` 내부에서 완전히 같은 dispatch 파이프라인을 공유하고, 사용자 입장에서도 "지도를 손으로 조작해 화면을 바꾸는 행위"라는 점에서 다르지 않다. 둘을 다르게 취급하면 특수 케이스만 늘어나 기각.
- **Kakao SDK의 `bounds_changed` 이벤트를 커밋 신호로 계속 사용한다**: 이 이벤트는 사용자 드래그와 프로그래매틱 `panTo` 양쪽 모두에서 발화하므로, 이벤트 종류만으로 "왜 bounds가 바뀌었는지"를 구분할 수 없다. 그래서 커밋은 이벤트에 의존하지 않고 세 지점(최초 로드, 명시적 지도 초점 이동 effect, 재검색 버튼 클릭)에서 직접 호출하도록 했다 — `bounds_changed`는 이제 track-only(dirty 표시)로만 쓰인다.
- **재검색 버튼을 검색창 옆(기존 위치)에 그대로 두고 ref로 지도 bounds에 접근한다**: `MapComponent`가 지도 객체를 캡슐화하고 있어 바깥(`page.tsx`)에서 실시간 bounds를 읽으려면 imperative handle을 새로 뚫어야 한다. 버튼을 지도 위 플로팅 오버레이로 옮기면 `MapComponent` 내부에서 `map` 객체에 직접 접근할 수 있어 배선이 단순해진다 — 네이버지도 등에서 흔한 패턴이기도 해 이쪽을 택함.

## Consequences

검색창 옆에 상시 노출되던 `RefreshSearchButton`(`page.tsx`)은 제거했다 — 카테고리 필터 변경은 이미 자동 재조회 대상이라 "수동 새로고침"이 필요한 유일한 시나리오(드래그/줌 이후)는 새 플로팅 버튼이 전담한다. `RegionSearchButton.tsx`의 프리셋(`CurrentRegionButton`, `MapBoundsSearchButton` 등) 자체는 그대로 남겨뒀다 — 재사용 가능한 UI 원자이고 다른 화면에서 쓸 수 있다.

마커 클릭은 이번 결정의 범위가 아니다 — `handleMarkerClick`은 원래도 영역 조회를 트리거하지 않고 `selectedMapMarker`만 바꾸며, `MemoizedMapMarker`가 `React.memo`로 감싸져 있어 선택이 바뀌어도 마커 1~2개만 재렌더링된다.
