# 활성 bounds RPC는 선택 필터를 행 단위 함수 대신 인라인한다

ADR-0004에서 category/search 술어를 SQL 함수로 공유했지만, 운영 데이터 실행계획에서는 대표 bounds의 공간 후보 8,039건마다 해당 함수가 호출되어 count 실행이 55.8ms까지 증가했다. 동등한 술어를 인라인하자 PostgreSQL이 `NULL` 필터를 제거하고 count는 6.6ms, 페이지와 총개수를 조립하는 전체 SQL의 3회 중앙값은 125.8ms에서 16.0ms로 감소했다. 따라서 프론트엔드가 사용하는 `get_reports_in_bounds_page`에서는 페이지와 총개수 조건을 중복하더라도 category/search 술어를 각각 인라인한다. 이 경로에서는 DRY보다 플래너가 파라미터와 데이터 선택도에 맞는 계획을 세울 수 있는 경계를 우선한다.

## Consequences

활성 RPC 안에 동일한 선택 필터가 두 번 존재하므로 SQL 계약 테스트로 두 술어의 동기화를 지킨다. 기존 주변 조회와 롤백용 bounds RPC의 공유 함수는 이번 변경에서 제거하지 않으며, 해당 경로를 다시 활성화할 때는 별도의 실행계획 검증이 필요하다.
